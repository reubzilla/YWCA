# ワークフローガイド

このガイドは、英語版を正本として同じ構成で作成しています。

## 参加可否の回答フロー

1. 教員が、有効かつ中止されていない活動を作成します。
2. 活動日が今後8週間以内になると、今後の活動の対象期間に表示されます。
3. 生徒または他の在籍中メンバーが参加可否を開きます。
4. 必要に応じて個人メモを入力し、`Available`、`Unavailable`、`Unsure`のいずれかを保存します。
5. サーバーが活動と回答を検証し、ロックを使用して本人のAvailability行を書き込みます。
6. 保存成功後、ホームの通知データが更新されます。
7. 活動当日になると、クラブリーダーと教員がダッシュボードで回答を確認できます。

**Coming Soon（近日公開）：** 参加可否保存時のResponse Deadlineの自動適用。

## 自分の担当を確認するフロー

1. クラブリーダーまたは教員が訪問担当を作成します。
2. 担当になった生徒またはクラブリーダーが、自分の訪問スケジュールを開きます。
3. サーバーがログイン中のメンバーを識別し、そのメンバーの担当だけを返します。
4. メンバーが活動、活動内容、場所、出発時刻、状態、メモを確認します。

## 訪問予定の管理フロー

1. クラブリーダーまたは教員が、有効な今後の活動を選びます。
2. 対象となる在籍中の生徒とクラブリーダー、および現在の参加可否を確認します。
3. 1名以上のメンバーを選び、担当内容を入力します。
4. サーバーが権限、活動、メンバー、状態、出発時刻、Session IDとMember IDの重複を検証します。
5. `Unavailable`の場合は管理者の確認が必要で、`Unsure`または未回答の場合は通知が表示されます。
6. ロックを使用して担当が保存され、本人の画面と当日のダッシュボードで利用できるようになります。
7. 安全に削除できない担当や過去の担当は`Cancelled`にし、安全な今後の担当だけを完全削除できます。

## 当日ダッシュボードのフロー

1. クラブリーダーまたは教員がダッシュボードを開きます。
2. サーバーが`Asia/Tokyo`の本日の日付で、有効かつ中止されていない活動を探します。
3. 選択した活動に、在籍中の生徒とクラブリーダー、Availability、Volunteer Assignments、Attendanceを組み合わせます。
4. 画面に集計、参加可否の区分、訪問担当、競合、出席集計が表示されます。
5. 参加可否の個人メモと出席情報は、権限のある管理者だけが確認できます。

**Coming Soon（近日公開）：** ポータルからのAttendance行の作成または編集。

## 活動のライフサイクル

1. 教員が活動を作成し、サーバーが読みやすく一意なSession IDを生成します。
2. 教員が対応している項目を編集できます。
3. 中止すると、活動種別`Cancelled`の無効な行として保持されます。
4. 完全削除の前にAvailability、Volunteer Assignments、Attendanceを確認します。
5. 関連レコードがある場合は削除が拒否されるため、中止して履歴を保持します。

## メンバーのライフサイクル

1. 教員がメンバーを作成し、サーバーが次の`M001`形式のMember IDを生成します。
2. 教員がプロフィール項目、ロール、日付、在籍状態を編集できます。
3. 無効化するとActiveがfalseになり、退会日が使用されます。
4. 完全削除の前に、関連するAvailability、Volunteer Assignments、Attendanceをすべて確認します。
5. 関連履歴がある場合は削除が拒否されるため、代わりに無効化します。
6. 管理権限を削除する前に、最後の在籍中教員と自己アクセスに関する保護が適用されます。

## 言語変更のフロー

1. ユーザーが日本語または英語を選びます。
2. 選択内容がブラウザのローカルストレージに保存されます。
3. 固定の画面テキストと現在の画面が、単一のローカライズ辞書から再描画されます。
4. 日付は`Asia/Tokyo`で、日本語またはイギリス英語の形式になります。
5. 内部値は英語のままで、氏名、活動名、メモ、活動内容、場所も変更されません。
