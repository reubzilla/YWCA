# UIリデザインのベースライン

この文書は、承認済みのUIリデザインを開始する前の、フロントエンドのルートと手動受け入れ基準を記録するものです。将来のナビゲーション案ではなく、現在のアプリケーションを説明しています。

英語版を基準文書とし、日本語版は同じ構成にします。

## Phase 4の現在の画面

以下のルート表は、回帰確認のためPhase 0時点の基準を残しています。現在のPhase 4ルート構成では、以下の内容が優先されます。

| ルートID | Student | Club Leader | Teacher | 現在の目的 |
|---|---:|---:|---:|---|
| `today` | 利用可 | 利用可 | 利用可 | Studentには対応優先の個人向け「本日」、Club LeaderとTeacherにはクラブ全体の運営画面 |
| `availability` | 利用可 | 利用可 | 利用可 | 個人の参加可否一覧と、選択した回答の編集画面 |
| `assignments` | 利用可 | 利用可 | 利用可 | ログイン中のメンバー本人の次の担当、今後の担当、担当履歴 |
| `notifications` | 利用可 | 利用可 | 利用可 | 個人向けの閲覧専用通知と、個人画面を開く操作 |
| `visitor-coordination` | 利用不可 | 利用可 | 利用可 | Club LeaderとTeacherによる訪問担当管理 |
| `planning` | 利用不可 | 利用可 | 利用可 | クラブ全体の今後の活動計画 |
| `sessions` | 利用不可 | 利用不可 | 利用可 | Teacher専用の活動・イベント管理 |
| `members` | 利用不可 | 利用不可 | 利用可 | Teacher専用のメンバー管理 |

Studentのモバイルナビゲーションは、本日、参加可否、担当、通知です。Club Leaderは、本日、訪問、計画、個人、その他です。個人の担当と参加可否は、訪問調整およびクラブ全体の計画とは別のルートです。

Phase 4では、以下を受け入れ条件に追加します。

- Studentの本日画面では、必要な対応を本日の活動や個人担当より先に表示する。
- 参加可否は全活動の完全なフォームを並べず、短い一覧と1件の編集画面を表示する。
- 未回答、未定、回答済みの順に並べ、各グループ内は日付順を維持する。
- 自分の担当を、次の担当、今後の担当、担当履歴に分ける。
- 通知から、回答通知は参加可否、担当通知は自分の担当を開く。
- Club Leaderの運営用ナビゲーションと個人用ナビゲーションを明確に分ける。
- Student向けAPIは、認証済み本人の参加可否、非公開メモ、担当だけを返す。

## 現在のフロントエンドルート

フロントエンドでルートを表示・非表示にすることは、認可境界ではありません。サーバー側のApps Script関数では、引き続きログイン中のメンバーを認証し、権限を個別に確認する必要があります。

| ルートID | 現在のナビゲーション表示 | Student | Club Leader | Teacher | 状態 | 現在の動作 |
|---|---|---:|---:|---:|---|---|
| `home` | ホーム | 利用可 | 利用可 | 利用可 | 動作済み | `getPortalData()`から、ログイン中のメンバー情報、個人向け通知、今後の活動を表示します。 |
| `availability` | 参加可否 | 利用可 | 利用可 | 利用可 | 動作済み | ログイン中のメンバーの今後の活動と参加可否を表示し、`saveAvailability()`で保存します。 |
| `volunteer` | 訪問スケジュール／訪問予定管理 | 利用可 | 利用可 | 利用可 | 動作済み・役割依存 | Studentには`getMyVolunteerAssignments()`から本人の割り当てのみを表示します。Club LeaderとTeacherには管理者用APIを使用した訪問予定管理を表示します。 |
| `attendance` | チェックイン | 利用可 | 利用可 | 利用可 | プレースホルダー | プレースホルダー文のみを表示します。生徒のチェックイン、手動出欠、QR出欠は未実装です。 |
| `dashboard` | ダッシュボード | 利用不可 | 利用可 | 利用可 | 動作済み・出欠は閲覧のみ | 本日のダッシュボードと今後の活動を表示します。ダッシュボードと今後の参加可否APIは、サーバー側で管理権限を確認します。出欠情報は表示されますが記録できません。 |
| `sessions` | 活動・イベント管理 | 利用不可 | 利用不可 | 利用可 | 動作済み | Teacher専用のサーバー関数を使用して活動を一覧・管理します。 |
| `members` | メンバー管理 | 利用不可 | 利用不可 | 利用可 | 動作済み | Teacher専用のサーバー関数を使用してメンバーを一覧・管理します。 |

## UIリデザイン受け入れチェックリスト

次の内容を含む代表的なデータを使用します。

- 回答期限が設定された活動を1件以上;
- Available、Unavailable、Unsure、未回答のメンバー;
- 訪問割り当てと、Unavailableのメンバーへの割り当て競合;
- 長い日本語および英語の活動名;
- 長いメンバー名、メモ、活動名、場所;
- 可能な範囲で、読み込み中、空、保存成功、入力エラー、サーバーエラーの各状態。

### デスクトップ — 幅1440px

- [ ] アプリが中央に配置され、ページ全体の横スクロールバーが表示されない。
- [ ] ヘッダー、言語選択、ナビゲーション、本文、フォーム、操作が読みやすい。
- [ ] 各役割で許可された現在の全ルートを開ける。
- [ ] 管理画面の一覧、フィルター、フォーム、集計値、詳細が重ならない。
- [ ] 長いユーザー入力およびスプレッドシートの文字列が、意図せず切れずに折り返される。
- [ ] すべての操作可能なコントロールでキーボードフォーカスが見える。

### ノートパソコン — 幅1024px

- [ ] ページ全体の横スクロールバーが表示されない。
- [ ] ナビゲーションが見つけやすく、表示された全項目に移動できる。
- [ ] 3列のフィルターとフォームで、ラベルやコントロールが切れずに操作できる。
- [ ] ダッシュボードの集計カードと今後の活動の件数表示が読みやすい。
- [ ] 主要、補助、削除系の操作を視覚的に区別できる。

### タブレット — 幅768px

- [ ] 縦向きと横向きの両方で使用できる。
- [ ] ナビゲーション、フィルター、フォーム、レコード操作が画面外にはみ出さない。
- [ ] 可能な範囲でタッチ対象が44×44 CSSピクセル以上である。
- [ ] 開く、保存、キャンセル、フィルター、更新をタッチで完了できる。
- [ ] 画面キーボードが入力中の項目や保存操作を継続的に隠さない。

### モバイル — 幅390px

- [ ] ページ全体の横スクロールバーが表示されない。
- [ ] ナビゲーション項目に到達でき、ラベルが動作を誤解させない。
- [ ] 参加可否の選択、非公開メモ、保存ボタンを操作できる。
- [ ] 管理操作が重ならずに縦に並び、優先順位が明確である。
- [ ] 日付、時刻、状態ラベル、氏名、メモ、場所が安全に折り返される。
- [ ] ブラウザー表示を200%にしても利用できる。

### 日本語

- [ ] 有効な保存設定がない場合、日本語が初期表示になる。
- [ ] ナビゲーション、見出し、コントロール、入力エラー、読み込み中、空状態、エラー、確認文が日本語辞書を使用する。
- [ ] 日付が日本語形式と`Asia/Tokyo`タイムゾーンで表示される。
- [ ] 日本語へ変更すると、参加可否の入力途中の内容を失わずに現在の画面が再描画される。
- [ ] 内部英語値、ユーザー入力、スプレッドシートの文字列が翻訳されない。

### 英語

- [ ] 英語を選択すると、ローカライズ対象の全UI文字列が更新される。
- [ ] `localStorage`により、再読み込み後も選択が維持される。
- [ ] 日付がイギリス英語形式と`Asia/Tokyo`タイムゾーンで表示される。
- [ ] 英語へ変更すると、参加可否の入力途中の内容を失わずに現在の画面が再描画される。
- [ ] 送信データの内部英語値が変更されない。

### Studentアカウント

- [ ] ホームに、ログイン中のStudent本人の個人情報、通知、今後の活動だけが表示される。
- [ ] 参加可否では、ログイン中のStudent本人の回答と非公開メモだけを読み書きする。
- [ ] 訪問スケジュールには、ログイン中のStudent本人の割り当てだけが表示される。
- [ ] ダッシュボード、活動管理、メンバー管理、管理者用訪問データを利用できない。
- [ ] チェックインが明確にプレースホルダーであり、出欠を記録したように見えない。

### Club Leaderアカウント

- [ ] ホームと個人の参加可否が正常に読み込まれる。
- [ ] 訪問予定管理で、現在の認可済み管理フローを使用できる。
- [ ] ダッシュボードの本日と今後の活動で、クラブ全体の管理データを読み込める。
- [ ] 参加可否の非公開メモは、認可済みの管理レスポンスでのみ表示される。
- [ ] 活動管理とメンバー管理を利用できない。
- [ ] 出欠は閲覧のみ、またはプレースホルダーのままである。

### Teacherアカウント

- [ ] ホームと個人の参加可否が正常に読み込まれる。
- [ ] ダッシュボードの本日と今後の活動を読み込める。
- [ ] 訪問予定管理が読み込まれ、既存の入力検証と競合警告が維持される。
- [ ] 活動・イベント管理が読み込まれ、作成、編集、キャンセル、保護された削除が維持される。
- [ ] メンバー管理が読み込まれ、役割、有効状態、最後のTeacher保護、保護された削除が維持される。
- [ ] レポートと設定が動作済みのフロントエンドルートとして表示されない。
- [ ] 出欠は閲覧のみ、またはプレースホルダーのままである。

## Phase 0の回帰境界

Phase 0では、ナビゲーション、スタイル、レイアウト、サーバーAPI、権限、シート構成、内部英語値を変更してはいけません。参加可否の回答期限テンプレートには、有効な段落要素が正確に1つだけ存在し、既存の参加可否操作は以前と同じように動作する必要があります。
